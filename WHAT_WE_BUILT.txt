================================================================================
THE PROBLEM WE'RE SOLVING
================================================================================

Companies use OpenAI/Anthropic APIs. They have no idea:
- How much they're spending (until the bill arrives)
- Which projects/teams are burning money
- What prompts are being called repeatedly (cache opportunity)
- When they're about to blow their budget

Our solution: A TRANSPARENT PROXY that sits between them and the LLM providers.

Every API call goes through us. We:
1. Track usage (tokens, cost, latency)
2. Write it to a durable log (WAL)
3. Forward the request like nothing happened

Later (Phase 2+), we'll:
- Drain the log into Postgres for queries
- Enforce budget caps (return 429 before they overspend)
- Cache identical requests in Redis
- Send daily Slack reports with spending breakdown

But for Phase 0+1, we just CAPTURE the data. That's it.

- zerolog: Structured logging (faster than zap, simpler than logrus)

WHY NOT USE A FRAMEWORK?
- Frameworks hide complexity. We want control.
- Frameworks add weight. We want lean.
- Frameworks change APIs. We want stability.

WHY "postgres" AS HOST?
- Docker Compose creates a DNS entry for each service name
- The proxy container can reach Postgres at hostname "postgres"
- In production, this would be an RDS endpoint

WHY sslmode=disable?
- In Docker Compose, traffic never leaves the host machine
- SSL adds latency for zero security benefit locally
- In production, you'd set this to "require" or "verify-full"

WHY STRUCTS INSTEAD OF MAP?
- Type safety. Typos caught at compile time.
- IDE autocomplete. You don't guess field names.
- Default values. Empty config file? We have sane defaults.

func Load(path string) (*Config, error) {
    // 1. Start with hardcoded defaults
    cfg := &Config{
        Server: ServerConfig{Host: "0.0.0.0", Port: 8080},
        Postgres: PostgresConfig{Host: "localhost", Port: 5432, ...},
        ...
    }

    // 2. If YAML exists, load it and merge
    if path != "" {
        data := os.ReadFile(path)
        yaml.Unmarshal(data, cfg)  // Overwrites defaults
    }

    // 3. Apply env var overrides (highest priority)
    applyEnvOverrides(cfg)

    return cfg
}

PRIORITY ORDER (highest to lowest):
1. Env vars (WITNESS_SERVER_PORT=9000)
2. YAML file (configs/proxy.yaml)
3. Hardcoded defaults (port: 8080)

WHY THIS ORDER?
- Env vars are how you override in production (Kubernetes secrets)
- YAML is how you configure locally (checked into git)
- Defaults are fallback (works out of the box)

WHY NOT VIPER/COBRA?
- Viper is 50KB of code for 20 lines of logic
- Cobra is for CLI apps (flags, subcommands). We're a daemon.
- Plain YAML + env vars is 100 lines. Easy to debug.

This is the Postgres schema. Embedded in the binary, runs on first boot.

CREATE TABLE IF NOT EXISTS projects (
    id              BIGSERIAL PRIMARY KEY,
    name            TEXT NOT NULL UNIQUE,
    slug            TEXT NOT NULL UNIQUE,
    timezone        TEXT NOT NULL DEFAULT 'UTC',
    report_hour     INT NOT NULL DEFAULT 9,
    cache_ttl_sec   INT NOT NULL DEFAULT 3600,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

WHY BIGSERIAL?
- Auto-incrementing 64-bit integer
- Postgres generates IDs (no coordination needed)
- Can handle 9 quintillion projects (you'll never run out)

WHY TIMESTAMPTZ?
- Stores timestamps in UTC, converts on read
- No timezone bugs (your customer's timezone ≠ your server's timezone)
- Always use TIMESTAMPTZ, never TIMESTAMP (trust me on this)

CREATE TABLE IF NOT EXISTS requests (
    id              BIGSERIAL PRIMARY KEY,
    project_id      BIGINT REFERENCES projects(id),
    provider        TEXT NOT NULL,          -- "openai" or "anthropic"
    model           TEXT NOT NULL,          -- "gpt-4o-mini"
    prompt_hash     TEXT NOT NULL,          -- SHA256 of request body (8 bytes)
    input_tokens    INT NOT NULL DEFAULT 0,
    output_tokens   INT NOT NULL DEFAULT 0,
    total_tokens    INT NOT NULL DEFAULT 0,
    cost            NUMERIC(12, 8) NOT NULL DEFAULT 0,  -- $0.00001234
    latency_ms      INT NOT NULL DEFAULT 0,
    status_code     INT NOT NULL DEFAULT 0,  -- 200, 429, 500, etc.
    cache_hit       BOOLEAN NOT NULL DEFAULT FALSE,
    stream          BOOLEAN NOT NULL DEFAULT FALSE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

WHY NUMERIC(12, 8)?
- 12 total digits, 8 after decimal
- Stores $9999.99999999 max
- No floating point errors (0.1 + 0.2 = 0.3, always)
- NEVER use FLOAT for money. EVER.

WHY prompt_hash?
- Phase 2: Detect duplicate prompts ("this prompt ran 1000 times yesterday")
- Cache key: If temperature=0 and prompt_hash matches, return cached response
- Privacy: We store hash, not the actual prompt text

CREATE TABLE IF NOT EXISTS prompts (
    id              BIGSERIAL PRIMARY KEY,
    project_id      BIGINT REFERENCES projects(id),
    prompt_hash     TEXT NOT NULL,
    sample_prefix   TEXT NOT NULL DEFAULT '',  -- First 100 chars (for debugging)
    total_calls     BIGINT NOT NULL DEFAULT 1,
    total_cost      NUMERIC(12, 8) NOT NULL DEFAULT 0,
    first_seen      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_seen       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (project_id, prompt_hash)
);

WHY THIS TABLE?
- Aggregates calls by prompt (1 row per unique prompt per project)
- Phase 4: "Your top 5 most expensive prompts cost $1,234 yesterday"
- ON CONFLICT DO UPDATE: increment total_calls, add to total_cost

CREATE INDEX IF NOT EXISTS idx_requests_project_created
    ON requests (project_id, created_at);

WHY THIS INDEX?
- Query: "Show me all requests for project X in the last 24 hours"
- Without index: Full table scan (slow, kills Postgres at 1M rows)
- With index: Seeks to project_id=X, scans only that range

CREATE INDEX IF NOT EXISTS idx_requests_prompt_hash
    ON requests (prompt_hash);

WHY?
- Query: "Find all requests with this prompt hash"
- Used by Phase 2 prompt aggregation

WHY NOT MORE INDEXES?
- Indexes slow down writes (every INSERT updates the index)
- Start with minimal indexes, add more based on actual queries
- Premature optimization is the root of all evil

package storage

import (
    _ "embed"
)

//go:embed schema.sql
var schemaSQL string

func Migrate(ctx context.Context, db *pgxpool.Pool) error {
    _, err := db.Exec(ctx, schemaSQL)
    return err
}

WHY EMBED?
- Schema is compiled into the binary (no external files to ship)
- Binary is self-contained (no "WHERE IS THE SCHEMA FILE?" bugs)
- Deployment: Just copy one binary. Done.

WHY NO MIGRATION FRAMEWORK?
- Frameworks (Flyway, Liquibase, golang-migrate) are for schema changes
- Phase 0-1: We create tables once, never change them
- Phase 2+: If we need migrations, we'll add versioning
- YAGNI: You Ain't Gonna Need It (yet)

WHEN WOULD WE ADD A MIGRATION FRAMEWORK?
- Phase 3: Adding columns (ALTER TABLE requests ADD COLUMN retry_count INT)
- Phase 4: Changing column types (ALTER ... TYPE ...)
- Phase 5: Adding partitioning (CREATE TABLE requests_2025_02 PARTITION OF ...)

For now? Run schema.sql on boot. If tables exist (IF NOT EXISTS), it's a no-op.

func main() {
    // 1. Init logging
    zerolog.TimeFieldFormat = zerolog.TimeFormatUnixMs
    log.Logger = zerolog.New(os.Stdout).With().Timestamp().Caller().Logger()

WHY ZEROLOG?
- JSON output: {"level":"info","time":1234567890,"caller":"main.go:42","msg":"starting server"}
- Parseable by log aggregators (Datadog, Splunk, CloudWatch)
- Structured: log.Info().Str("addr", ":8080").Msg("listening")
- Fast: Zero allocations for disabled log levels

WHY JSON LOGS?
- Text logs: "2025-01-20 10:30:45 INFO Starting server on :8080"
  - Hard to parse programmatically
  - No structured data (can't filter by field)

- JSON logs: {"level":"info","time":1234567890,"addr":":8080","msg":"starting"}
  - Every field is queryable in log aggregator
  - Can alert on specific errors: level="error" AND msg="postgres connection failed"

    // 2. Load config
    cfgPath := os.Getenv("WITNESS_CONFIG")
    if cfgPath == "" {
        cfgPath = "configs/proxy.yaml"
    }
    cfg, err := config.Load(cfgPath)

FLOW:
1. Check env var WITNESS_CONFIG
2. If not set, use configs/proxy.yaml
3. If that doesn't exist, use hardcoded defaults
4. Apply env var overrides (WITNESS_POSTGRES_HOST, etc.)

    // 3. Connect to Postgres
    pgPool, err := pgxpool.New(ctx, cfg.Postgres.DSN())
    if err != nil {
        log.Fatal().Err(err).Msg("failed to connect to postgres")
    }
    defer pgPool.Close()

WHY pgxpool?
- Connection pool: Reuses TCP connections (don't pay handshake cost every request)
- MaxConns: 10 by default (can handle 100+ req/s per connection)
- MinConns: 2 by default (keeps connections warm)
- Health check: Closes stale connections automatically

WHY Fatal() ON ERROR?
- If Postgres is down, the app can't function
- Don't start the HTTP server if dependencies are broken
- Fail fast, fail loudly (easier to debug in production)

    if err := pgPool.Ping(ctx); err != nil {
        log.Fatal().Err(err).Msg("failed to ping postgres")
    }

WHY PING?
- pgxpool.New() creates the pool but doesn't connect yet (lazy)
- Ping() forces a connection attempt
- If Postgres is down, we fail here (not 30 seconds later on first request)

    // 4. Run migrations
    if err := storage.Migrate(ctx, pgPool); err != nil {
        log.Fatal().Err(err).Msg("failed to run migrations")
    }

WHAT THIS DOES:
- Runs schema.sql (CREATE TABLE IF NOT EXISTS ...)
- Idempotent: Safe to run every boot
- Phase 2+: This is where we'd apply versioned migrations

    // 5. Connect to Redis
    rdb := redis.NewClient(&redis.Options{
        Addr:     cfg.Redis.Addr(),  // "redis:6379"
        Password: cfg.Redis.Password,
        DB:       cfg.Redis.DB,
    })
    defer rdb.Close()

    if err := rdb.Ping(ctx).Err(); err != nil {
        log.Fatal().Err(err).Msg("failed to ping redis")
    }

WHY REDIS NOW?
- Not used in Phase 0-1, but we verify it's reachable
- Phase 3: This connection will be used for cache + budgets
- Fail fast: If Redis is misconfigured, find out at boot (not 2 hours later)

    // 6. Set up router
    r := chi.NewRouter()
    r.Use(middleware.RealIP)
    r.Use(middleware.RequestID)
    r.Use(metricsMiddleware)

WHY CHI?
- Lightweight (faster than gin, cleaner than stdlib)
- Context-based (passes values via context.Context, not globals)
- HTTP 1.1 + HTTP/2 support out of the box

middleware.RealIP:
- Extracts client IP from X-Forwarded-For header
- When behind a load balancer, the direct connection IP is the LB's IP
- X-Forwarded-For: 203.0.113.42, 10.0.0.1 (client IP is 203.0.113.42)

middleware.RequestID:
- Generates unique ID for each request (X-Request-ID: abc-123-def-456)
- Logs include request_id field (can trace a request through logs)
- If client sends X-Request-ID, we reuse it (distributed tracing)

metricsMiddleware:
- Increments prometheus counters: witness_http_requests_total{method,status}
- Records latency histogram: witness_http_request_duration_seconds{method}

    r.Get("/healthz", func(w http.ResponseWriter, r *http.Request) {
        // Check Postgres
        if err := pgPool.Ping(r.Context()); err != nil {
            http.Error(w, `{"status":"unhealthy","error":"postgres"}`, 503)
            return
        }
        // Check Redis
        if err := rdb.Ping(r.Context()).Err(); err != nil {
            http.Error(w, `{"status":"unhealthy","error":"redis"}`, 503)
            return
        }
        w.Header().Set("Content-Type", "application/json")
        w.WriteHeader(200)
        fmt.Fprint(w, `{"status":"healthy"}`)
    })

WHY THIS ENDPOINT?
- Load balancers call /healthz every 5 seconds
- If it returns non-200, LB removes instance from rotation
- We verify dependencies (Postgres, Redis) are reachable

WHY 503 FOR UNHEALTHY?
- 503 = Service Unavailable (temporary failure)
- Load balancer sees 503 → stops sending traffic → instance recovers
- If we returned 200 when unhealthy, LB would keep sending traffic → requests fail

    r.Handle("/metrics", promhttp.Handler())

WHAT THIS DOES:
- Exposes Prometheus metrics in text format
- Scraped by Prometheus server every 15 seconds
- Powers Grafana dashboards (Phase 5)

    // 7. Graceful shutdown
    srv := &http.Server{
        Addr:         ":8080",
        Handler:      r,
        ReadTimeout:  30 * time.Second,   // How long to read request body
        WriteTimeout: 120 * time.Second,  // How long to write response (streaming!)
        IdleTimeout:  120 * time.Second,  // How long to keep connection alive
    }

    sigCh := make(chan os.Signal, 1)
    signal.Notify(sigCh, syscall.SIGINT, syscall.SIGTERM)

    go func() {
        log.Info().Str("addr", ":8080").Msg("starting server")
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatal().Err(err).Msg("server error")
        }
    }()

    sig := <-sigCh  // Block here until signal received
    log.Info().Str("signal", sig.String()).Msg("shutting down")

    shutdownCtx, cancel := context.WithTimeout(context.Background(), 15*time.Second)
    defer cancel()

    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Error().Err(err).Msg("shutdown error")
    }

WHY GRACEFUL SHUTDOWN?
- When Kubernetes kills the pod, it sends SIGTERM
- Without graceful shutdown: In-flight requests get TCP RST (connection reset)
- With graceful shutdown:
  1. Stop accepting new requests
  2. Wait for in-flight requests to finish (up to 15 seconds)
  3. Close connections cleanly
  4. Exit

WHY 15 SECOND TIMEOUT?
- Kubernetes default terminationGracePeriodSeconds is 30s
- We give requests 15s to finish
- Kubernetes gives us 30s total → plenty of buffer

WHY 120s WRITE TIMEOUT?
- Streaming responses can be LONG (GPT-4 reasoning might take 60+ seconds)
- 30s would kill streaming requests mid-response
- 120s is generous but safe

- Statically linked binary (no libc dependency)

- -s: Strip symbol table (debug symbols)
- -w: Strip DWARF debugging info
- Reduces binary from 30MB → 21MB (30% smaller)
- You can't attach a debugger, but you have logs + metrics

THE ARCHITECTURE (BIG PICTURE)
------------------------------------------------------------------------

    Client
      |
      | POST /openai/v1/chat/completions
      | Authorization: Bearer sk-xxx
      | X-Project: my-app
      | {"model":"gpt-4o-mini","messages":[...]}
      |
      v
    PROXY (our code)
      |
      |-- 1. Resolve project: X-Project header → "my-app"
      |                        (or SHA256(Authorization) → "key:a1b2c3d4")
      |                        (or fallback → "unknown")
      |
      |-- 2. Detect streaming: Is "stream":true in body?
      |       YES → Go to streaming path
      |       NO  → Go to non-streaming path
      |
      |-- 3. Forward to provider:
      |       Strip /openai prefix → forward to https://api.openai.com/v1/...
      |       Copy headers (Authorization, etc.)
      |       Remove X-Project (internal header, don't leak it)
      |
      |-- 4. Get response from provider
      |
      |-- 5. Extract usage:
      |       Non-streaming: Parse response JSON → usage.prompt_tokens
      |       Streaming: Accumulate SSE events → find final usage chunk
      |
      |-- 6. Compute cost:
      |       model="gpt-4o-mini" → $0.15/1M input, $0.60/1M output
      |       10 input tokens + 5 output tokens = $0.0000045
      |
      |-- 7. Write to WAL (BEFORE returning response!)
      |       Append to /data/wal/wal-2025-01-20.jsonl
      |       fsync() to disk (durability guarantee)
      |
      |-- 8. Return response to client
      |       Streaming: Forward SSE events (but HIDE usage-only chunk)
      |       Non-streaming: Forward JSON response as-is
      |
      v
    Client receives response (unaware we were in the middle)

This file defines the PROVIDER ABSTRACTION.

type Usage struct {
    InputTokens  int
    OutputTokens int
    TotalTokens  int
    Model        string
}

WHY?
- Every provider (OpenAI, Anthropic, Gemini) returns usage differently
- We normalize to this struct
- Rest of the code doesn't care which provider

type Provider struct {
    Name       string                                    // "openai"
    PathPrefix string                                    // "/openai"
    Target     *url.URL                                  // https://api.openai.com
    ExtractUsage       func(body []byte) (Usage, error) // Parse JSON response
    ExtractStreamUsage func(events []SSEEvent) (Usage, error)
    IsStreamRequest    func(body []byte) bool            // Check if "stream":true
    PrepareStreamBody  func(body []byte) ([]byte, error) // Modify request before forward
}

WHY FUNCTION FIELDS?
- Each provider has different logic
- OpenAI: response.usage.prompt_tokens
- Anthropic: response.usage.input_tokens (different field name!)
- Gemini: No usage in streams (need to estimate or call /countTokens)

WHY PrepareStreamBody?
- OpenAI needs "stream_options":{"include_usage":true} injected
- Anthropic doesn't need this (usage is always in stream)
- Provider-specific logic, encapsulated here

var Registry = map[string]*Provider{}

func Register(p *Provider) {
    Registry[p.PathPrefix] = p
}

func Lookup(path string) *Provider {
    for prefix, p := range Registry {
        if len(path) >= len(prefix) && path[:len(prefix)] == prefix {
            return p
        }
    }
    return nil
}

HOW IT WORKS:
1. openai.go calls Register() in init() → adds /openai to registry
2. anthropic.go calls Register() in init() → adds /anthropic to registry
3. Request comes in: /openai/v1/chat/completions
4. Lookup("/openai/v1/chat/completions") → returns openai provider

func extractOpenAIStreamUsage(events []SSEEvent) (Usage, error) {
    var usage Usage
    // Walk events in REVERSE (usage is in the last chunk)
    for i := len(events) - 1; i >= 0; i-- {
        ev := events[i]
        if ev.Data == "" || ev.Data == "[DONE]" {
            continue
        }
        var chunk openaiStreamChunk
        if err := json.Unmarshal([]byte(ev.Data), &chunk); err != nil {
            continue  // Malformed chunk, skip
        }
        if chunk.Model != "" {
            usage.Model = chunk.Model
        }
        if chunk.Usage != nil {
            usage.InputTokens = chunk.Usage.PromptTokens
            usage.OutputTokens = chunk.Usage.CompletionTokens
            usage.TotalTokens = chunk.Usage.TotalTokens
            return usage, nil  // FOUND IT, STOP
        }
    }
    // No usage found (stream_options not injected?), return model only
    return usage, nil
}

WHY REVERSE ITERATION?
- Streaming response has 10-100 chunks
- Usage is in the LAST chunk
- Going forward → parse 99 chunks before finding usage
- Going backward → parse 1 chunk, find usage immediately

func prepareOpenAIStreamBody(body []byte) ([]byte, error) {
    var req map[string]any
    json.Unmarshal(body, &req)

    // Inject stream_options
    streamOpts, ok := req["stream_options"].(map[string]any)
    if !ok {
        streamOpts = map[string]any{}
    }
    streamOpts["include_usage"] = true
    req["stream_options"] = streamOpts

    return json.Marshal(req)
}

WHAT THIS DOES:
    BEFORE: {"model":"gpt-4o-mini","stream":true,"messages":[...]}
    AFTER:  {"model":"gpt-4o-mini","stream":true,"stream_options":{"include_usage":true},"messages":[...]}

- By default, OpenAI streaming responses DON'T include usage
- "stream_options":{"include_usage":true} tells OpenAI to add the usage chunk

- Regex is brittle: What if there's already a stream_options field?
- JSON decode → modify → encode is safe and correct
- Slightly slower (3µs overhead), but correctness > speed

    var usage Usage

    for _, ev := range events {
        if ev.Data == "" {
            continue
        }

        switch ev.Event {
        case "message_start":
            // Parse input tokens from initial event
            var msg anthropicStreamMessageStart
            json.Unmarshal([]byte(ev.Data), &msg)
            if msg.Message != nil {
                usage.Model = msg.Message.Model
                if msg.Message.Usage != nil {

                }

        case "message_delta":

            }
        }
    }

    usage.TotalTokens = usage.InputTokens + usage.OutputTokens
    return usage, nil
}

func ResolveProject(r *http.Request) string {
    // Priority 1: X-Project header
    if project := r.Header.Get("X-Project"); project != "" {
        return project
    }

    // Priority 2: SHA256(Authorization header)
    if auth := r.Header.Get("Authorization"); auth != "" {
        hash := sha256.Sum256([]byte(auth))
        return fmt.Sprintf("key:%x", hash[:16])  // 16 bytes = 32 hex chars
    }

    // Priority 3: Unknown
    return "unknown"
}

WHY THIS PRIORITY?
1. X-Project is explicit (customer tells us which project)
2. Authorization hash is implicit (stable per API key)
3. "unknown" is fallback (should be rare)

WHY SHA256(Authorization)?
- Same API key → same project identifier (deterministic)
- Different API keys → different project identifiers (unique)
- Hash is one-way (can't reverse to get the API key)

EXAMPLE:
    Authorization: Bearer sk-test-abc123
    SHA256("Bearer sk-test-abc123") = a1b2c3d4e5f6g7h8...
    project = "key:a1b2c3d4e5f6g7h8"

WHY ONLY 16 BYTES?
- 16 bytes = 128 bits of entropy (2^128 combinations)
- Collision probability: ~0% for millions of API keys
- Full 32 bytes (256 bits) is overkill (wastes DB space)

type Writer struct {
    dir      string
    mu       sync.Mutex
    file     *os.File
    fileDate string
    pending  int
    done     chan struct{}
    closeOnce sync.Once
}

WHY sync.Mutex?
- Multiple goroutines can call Write() concurrently
- Mutex ensures only one goroutine appends at a time
- Without mutex: Writes would interleave → corrupted JSONL

WHY closeOnce?
- Close() can be called multiple times (defer + explicit call)
- Without sync.Once: Second close() would panic (closing closed channel)
- With sync.Once: Second close() is a no-op

func NewWriter(dir string) (*Writer, error) {
    os.MkdirAll(dir, 0755)  // Create directory if not exists
    w := &Writer{
        dir:  dir,
        done: make(chan struct{}),
    }
    go w.syncLoop()  // Background goroutine for batched fsync
    return w, nil
}

WHY BACKGROUND GOROUTINE?
- Fsync every 100ms (even if < 50 records)
- Ensures records hit disk within 100ms worst-case
- Trade-off: 100ms latency vs. 1ms (fsync every write)

func (w *Writer) Write(rec Record) error {
    rec.Time = time.Now().UTC()  // Always UTC, always millisecond precision

    line, err := json.Marshal(rec)
    if err != nil {
        return err
    }
    line = append(line, '\n')  // JSONL format: one record per line

    w.mu.Lock()
    defer w.mu.Unlock()

    // Ensure file is open (creates new file if date changed)
    if err := w.ensureFile(); err != nil {
        return err
    }

    // Append to file
    if _, err := w.file.Write(line); err != nil {
        return err
    }

    w.pending++

    // Fsync every 50 records (throughput optimization)
    if w.pending >= 50 {
        if err := w.file.Sync(); err != nil {
            return err
        }
        w.pending = 0
    }

    return nil
}

THE FSYNC GUARANTEE (CRITICAL):
- Without fsync: Data is in kernel buffer, not on disk
- If kernel crashes, data is lost (rare but happens)
- With fsync: Data is on physical disk, durable
- Cost: ~0.5ms for HDD, ~0.05ms for SSD

BATCHING STRATEGY:
- Fsync every 50 records: Amortizes cost over 50 writes
- 50 writes * 0.05ms fsync = 0.001ms per write (negligible)
- Background loop: Fsyncs every 100ms if pending > 0
- Result: Max 100ms latency, <1ms average overhead

WHY FSYNC BEFORE RESPONSE?
- If we return response THEN fsync:
  - Client thinks request succeeded
  - Kernel crashes before fsync
  - WAL doesn't have the record
  - Reconciliation: Customer was charged by OpenAI, we have no record
  - Customer: "Where's my money going?!"
  - You: "Uhh... we lost your data"
  - Customer: Fires you, switches to competitor

- If we fsync THEN return response:
  - Kernel crashes after fsync
  - WAL has the record (durable)
  - Reconciliation: WAL matches OpenAI bill ✅
  - Customer: Happy, renews contract

func (w *Writer) ensureFile() error {
    today := time.Now().UTC().Format("2006-01-02")  // "2025-01-20"
    if w.file != nil && w.fileDate == today {
        return nil  // File is still valid
    }

    // Date changed (midnight UTC) or first boot
    if w.file != nil {
        w.file.Sync()   // Flush old file
        w.file.Close()
        w.pending = 0
    }

    path := filepath.Join(w.dir, fmt.Sprintf("wal-%s.jsonl", today))
    f, err := os.OpenFile(path, os.O_CREATE|os.O_APPEND|os.O_WRONLY, 0644)
    if err != nil {
        return err
    }

    w.file = f
    w.fileDate = today
    return nil
}

FILE ROTATION:
- One file per day: wal-2025-01-20.jsonl, wal-2025-01-21.jsonl, ...
- At midnight UTC, new file is created
- Old file is closed (but kept on disk)
- Phase 2: Drain worker processes old files
- Phase 5: Archive old files to S3 after 7 days

WHY DAILY ROTATION?
- Files don't grow unbounded (max 1 day of data per file)
- Drain worker can process completed days in parallel
- Easy to archive: "Move all wal-*.jsonl older than 7 days to S3"

func (w *Writer) syncLoop() {
    ticker := time.NewTicker(100 * time.Millisecond)
    defer ticker.Stop()
    for {
        select {
        case <-w.done:
            return  // Close() was called, exit
        case <-ticker.C:
            w.mu.Lock()
            if w.file != nil && w.pending > 0 {
                w.file.Sync()
                w.pending = 0
            }
            w.mu.Unlock()
        }
    }
}

WHAT THIS DOES:
- Every 100ms, check if there are pending writes
- If yes, fsync to disk
- If no, do nothing

WHY 100ms?
- 10 fsyncs per second = 0.5ms total overhead per second
- Max latency: 100ms (if record arrives right after tick)
- Average latency: 50ms (assuming uniform distribution)

func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    provider := providers.Lookup(r.URL.Path)
    if provider == nil {
        http.Error(w, `{"error":"unknown provider"}`, 404)
        return
    }

    // Read request body
    body, err := io.ReadAll(r.Body)
    if err != nil {
        http.Error(w, `{"error":"failed to read request body"}`, 400)
        return
    }
    r.Body.Close()

    // Resolve attribution
    project := ResolveProject(r)

    // Check if streaming
    isStream := provider.IsStreamRequest != nil && provider.IsStreamRequest(body)

    if isStream {
        h.handleStream(w, r, provider, body, project)
        return
    }

    h.handleNonStream(w, r, provider, body, project)
}

PATH STRIPPING:
    Client sends:     /openai/v1/chat/completions
    We strip:         /openai
    Upstream sees:    /v1/chat/completions
    Upstream URL:     https://api.openai.com/v1/chat/completions

func (h *Handler) handleNonStream(...) {
    start := time.Now()

    // Build upstream URL
    upstreamPath := strings.TrimPrefix(r.URL.Path, provider.PathPrefix)
    upstreamURL := provider.Target.String() + upstreamPath
    if r.URL.RawQuery != "" {
        upstreamURL += "?" + r.URL.RawQuery
    }

    // Create upstream request
    upReq, _ := http.NewRequestWithContext(r.Context(), r.Method, upstreamURL, bytes.NewReader(body))

    // Copy headers (Authorization, Content-Type, etc.)
    copyHeaders(r.Header, upReq.Header)
    upReq.Header.Set("Content-Length", fmt.Sprintf("%d", len(body)))

    // Forward to upstream
    resp, err := h.Transport.RoundTrip(upReq)
    if err != nil {
        http.Error(w, `{"error":"upstream request failed"}`, 502)
        return
    }
    defer resp.Body.Close()

    // Read response
    respBody, _ := io.ReadAll(resp.Body)
    latency := time.Since(start)

    // Extract usage
    var usage providers.Usage
    if resp.StatusCode >= 200 && resp.StatusCode < 300 {
        usage, _ = provider.ExtractUsage(respBody)
    }

    // Compute cost
    cost := providers.ComputeCost(usage.Model, usage.InputTokens, usage.OutputTokens)

    // Hash prompt
    promptHash := hashPrompt(body)

    // ⚠️ CRITICAL: Write WAL BEFORE returning response ⚠️
    h.WAL.Write(wal.Record{
        Project:      project,
        Provider:     provider.Name,
        Model:        usage.Model,
        PromptHash:   promptHash,
        InputTokens:  usage.InputTokens,
        OutputTokens: usage.OutputTokens,
        TotalTokens:  usage.TotalTokens,
        Cost:         cost,
        LatencyMs:    latency.Milliseconds(),
        StatusCode:   resp.StatusCode,
        CacheHit:     false,
        Stream:       false,
    })

    // Write response to client
    copyHeaders(resp.Header, w.Header())
    w.WriteHeader(resp.StatusCode)
    w.Write(respBody)
}

THE ORDER MATTERS:
    1. Forward to OpenAI
    2. Get response
    3. Extract usage
    4. Compute cost
    5. Write to WAL (FSYNC)   ← BLOCKS HERE
    6. Return response to client

If we did 6 before 5, and the server crashed at step 6, we'd lose the record.

func (h *Handler) handleStream(...) {
    // Prepare body (inject stream_options for OpenAI)
    if provider.PrepareStreamBody != nil {
        body, _ = provider.PrepareStreamBody(body)
    }

    // Forward to upstream (same as non-streaming)
    upstreamURL := ...
    upReq, _ := http.NewRequestWithContext(...)
    resp, _ := h.Transport.RoundTrip(upReq)

    // If upstream returned non-SSE (error), pass through
    contentType := resp.Header.Get("Content-Type")
    if !strings.Contains(contentType, "text/event-stream") {
        respBody, _ := io.ReadAll(resp.Body)
        w.WriteHeader(resp.StatusCode)
        w.Write(respBody)
        return
    }

    // Set up streaming response to client
    flusher := w.(http.Flusher)
    w.Header().Set("Content-Type", "text/event-stream")
    w.WriteHeader(resp.StatusCode)

    // Accumulate events while streaming to client
    var events []providers.SSEEvent
    scanner := bufio.NewScanner(resp.Body)
    scanner.Buffer(make([]byte, 0, 256*1024), 1024*1024)  // 256KB initial, 1MB max

    var currentEvent providers.SSEEvent

    for scanner.Scan() {
        line := scanner.Text()

        if line == "" {
            // Empty line = end of event
            if currentEvent.Data != "" || currentEvent.Event != "" {
                events = append(events, currentEvent)

                // Check if this is usage-only chunk (OpenAI)
                isUsageChunk := false
                if provider.Name == "openai" && currentEvent.Data != "[DONE]" {
                    isUsageChunk = isOpenAIUsageOnlyChunk(currentEvent.Data)
                }

                if !isUsageChunk {
                    // Forward event to client
                    writeSSEEvent(w, currentEvent)
                    flusher.Flush()
                }

                currentEvent = providers.SSEEvent{}
            }
            continue
        }

        if strings.HasPrefix(line, "event:") {
            currentEvent.Event = strings.TrimSpace(strings.TrimPrefix(line, "event:"))
        } else if strings.HasPrefix(line, "data:") {
            data := strings.TrimPrefix(line, "data:")
            data = strings.TrimPrefix(data, " ")
            if currentEvent.Data != "" {
                currentEvent.Data += "\n" + data  // Multi-line data
            } else {
                currentEvent.Data = data
            }
        }
    }

    latency := time.Since(start)

    // Extract usage from accumulated events
    usage, _ := provider.ExtractStreamUsage(events)
    cost := providers.ComputeCost(usage.Model, usage.InputTokens, usage.OutputTokens)

    // ⚠️ Write WAL BEFORE connection closes ⚠️
    h.WAL.Write(wal.Record{
        ...
        Stream: true,
    })
}

WHY ACCUMULATE EVENTS?
- We need ALL events to extract usage (it's at the end)
- We can't wait to forward (client wants immediate streaming)
- Solution: Accumulate in memory, forward in real-time

MEMORY USAGE:
- Typical streaming response: 50-100 events
- Each event: ~200 bytes
- Total: 10-20KB per request
- Max concurrent streams: 1000
- Peak memory: 20MB (negligible)

WHY FILTER USAGE-ONLY CHUNK?
- OpenAI SDK doesn't expect this chunk (we injected stream_options)
- If we forward it: SDK sees unknown chunk → logs warning → customer confused
- If we filter it: SDK works normally, we capture usage silently

USAGE-ONLY CHUNK:
    data: {"id":"c","model":"gpt-4o-mini","choices":[],"usage":{"prompt_tokens":10,"completion_tokens":2,"total_tokens":12}}

    choices=[] (no delta) + usage present = USAGE-ONLY

func isOpenAIUsageOnlyChunk(data string) bool {
    var chunk struct {
        Usage   *json.RawMessage
        Choices []struct {
            Delta struct {
                Content *string
            }
        }
    }
    json.Unmarshal([]byte(data), &chunk)

    if chunk.Usage == nil {
        return false  // No usage
    }
    if len(chunk.Choices) == 0 {
        return true  // Usage + no choices = usage-only
    }
    for _, c := range chunk.Choices {
        if c.Delta.Content != nil && *c.Delta.Content != "" {
            return false  // Has content
        }
    }
    return true  // Usage + empty/null content = usage-only
}

   → NO. Every LLM provider uses HTTP/SSE
   → gRPC would require protocol translation (adds latency)
   → HTTP is the right choice

4. Use TypeScript/Node.js instead of Go?
   → NO. Node.js is slower (V8 JIT vs. compiled)
   → Goroutines are lighter than threads (10K concurrent streams is easy)
   → Go binary is self-contained (Node needs runtime + node_modules)

TOTAL: 216.5µs ≈ 240µs (rounding + syscall overhead)
